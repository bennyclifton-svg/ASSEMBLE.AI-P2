## What to build

Create the first weekly briefing/reporting tracer: it reads current project records, typed RFIs, documents/RAG excerpts, risks, cost/programme context, correspondence where available, and approved memory, then produces a grounded report draft that distinguishes facts, assumptions, recommendations, and citations.

- [x] The workflow uses structured project records and stored documents rather than chat transcript as project truth.
- [x] The draft includes at least project status summary, current RFIs, programme/cost/risk highlights where available, open questions, and recommended actions.
- [x] Material claims cite project records or document excerpts where available.
- [x] The draft distinguishes facts, assumptions, and recommendations.
- [x] The draft can be reviewed before becoming an issued/report artefact.
- [x] Tests cover the workflow boundary with fake model output and real seeded project records, including at least one typed RFI.

## Agent Brief

**Category:** enhancement
**Summary:** Add the first weekly briefing/reporting workflow that drafts a grounded, reviewable report from project records, typed RFIs, evidence, and approved memory.

**Current behavior after implementation:**
Sitewise now has the weekly reporting tracer in addition to the underlying project records, typed RFIs, RFI evidence/export history, reports with sections and reporting periods, reviewable AI memory, approval-gated actions, project refresh events, RAG/document retrieval, and shared AI context assembly.

The weekly draft path is implemented as `correspondence.weekly_report.create_draft` / `create_weekly_report_draft`, with a workspace API route and a Reports panel command. It assembles structured project records, typed RFIs, document/RAG excerpts, and approved AI memory, then saves an editable report draft whose sections separate facts, assumptions, recommendations, open questions, and citations. Chat transcript is explicitly not project truth for this path.

**Desired behavior:**
A PM should be able to start a weekly briefing/report draft workflow from the project workspace or chat/workflow surface. The workflow should assemble current project truth from structured records and stored files: project details/profile/objectives, typed RFIs, cost plan and variations, programme and milestones, risks, relevant notes/correspondence/reports where available, document/RAG excerpts, and approved AI memory as advisory context only.

The workflow should create a reviewable report draft rather than an issued artefact. The draft must clearly separate factual status, assumptions, PM-review questions, recommendations, and citations/evidence. Material factual claims should cite project records or document excerpts where available, and unsupported assumptions should be labelled as assumptions instead of being presented as facts. The workflow may ask one concise branch-setting question at a time when PM judgement is genuinely required, with a recommended answer, and accepted judgement should become either approved memory or an approved record change only through registered actions and approval controls.

When the draft is saved, it should land in the existing project report/reports model as a draft or reviewable report with sections that a PM can edit before issuing/exporting. No email sending, external issuance, or silent project-data mutation should occur as part of this tracer.

**Key interfaces:**
- `assembleContext()` / `AssembledContext` - should support a weekly briefing/reporting task with current records, document/RAG evidence, cross-module insights, and approved AI memory clearly separated from canonical facts.
- Report records and report sections - should store the generated draft as a reviewable project report with a reporting period and editable section content.
- Typed RFI read model - should provide current RFI status, responsibility, due/overdue state, response state, and evidence references suitable for report drafting.
- AI memory entries - should be included only as labelled advisory preferences/context and must not override schema facts or document evidence.
- Workflow plan/run model - if the tracer is exposed as a workflow, it should have a durable run, execution brief, evidence summary, and registered action-backed write steps.
- `create_report` or equivalent report action - report creation or report-section writes initiated by the agent/workflow should use registered actions and explicit approval/proposal behavior where required by policy.
- Report draft generator/model client - should be injectable or fakeable so tests can verify grounding, citations, assumptions, and section output without depending on live model wording.

**Acceptance criteria:**
- [x] A PM can start a weekly briefing/report draft flow for a project and reporting period from the project workspace or chat/workflow entry point.
- [x] The flow reads structured project records and stored documents/RAG excerpts as project truth, not chat transcript.
- [x] The draft includes at least project status summary, current RFIs, programme highlights, cost/variation highlights, risk highlights, open questions, and recommended actions where data exists.
- [x] Material factual claims cite project records, typed RFIs, issued/export records, or document/RAG excerpts when available.
- [x] The draft separates facts, assumptions, recommendations, and PM-review questions in a way the UI or report section content preserves.
- [x] Approved AI memory can influence style or recurring context only as labelled advisory context and loses to schema/document evidence on conflicts.
- [x] The generated report remains reviewable/editable and is not treated as issued or externally sent by this workflow.
- [x] Any agent/workflow-created report, report-section update, memory write, or record write goes through registered actions and approval/audit behavior required by the action-only policy.
- [x] Tests cover the workflow/generator boundary with fake model output and seeded project records, including at least one typed RFI, one cited document/RAG excerpt, one assumption, and one recommendation.
- [x] Tests cover a conflict case where memory disagrees with a schema/document fact and the draft follows the canonical record or flags the memory as advisory/conflicting.

**Out of scope:**
- Issuing, emailing, or externally sending the weekly report.
- Full report export/version unification beyond using existing report draft/edit surfaces.
- Replacing the existing project-brief interview feature.
- Building new typed registers for EOTs, defects, site instructions, progress claims, determinations, or notices.
- Broad replacement of all direct UI report editing routes.
- Cross-project reporting, portfolio reporting, proactive monitoring, or scheduled automatic report generation.

## Implementation Notes

- Added `src/lib/weekly-report-draft/service.ts` to assemble structured project context, typed RFIs, document/RAG excerpts, and approved AI memory into a source register for weekly report drafting.
- Added registered action `correspondence.weekly_report.create_draft` with tool name `create_weekly_report_draft`; user-initiated calls run directly and agent/workflow calls produce proposals through the Action Registry.
- Added `/api/projects/[projectId]/weekly-report-draft` and a Reports panel "Weekly draft" command so PMs can start the flow from the project workspace.
- Draft output is saved as an editable report with custom sections for project status summary, current RFIs, programme/cost/risk highlights, open questions, and recommended actions.
- The generator normalizes model output into Facts, Assumptions, Recommendations, and Open Questions, preserving citation ids in section content.
- Approved AI memory is passed as advisory context only; memory-only factual claims are moved into assumptions instead of being treated as record-backed facts.
- Focused tests cover the action preview/apply path, the API route, seeded typed-RFI/document-memory grounding, and the memory-conflict boundary.
