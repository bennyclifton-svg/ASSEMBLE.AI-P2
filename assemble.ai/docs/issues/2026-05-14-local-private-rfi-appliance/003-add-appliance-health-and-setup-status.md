## Agent Brief

**Category:** enhancement
**Summary:** Add a PM-readable appliance health API and setup/status surface for local/private Sitewise instances.

**Current behavior:**
The app has a narrow health endpoint that mainly reports application/database reachability, while the broader local/private readiness checks live in a developer-facing smoke command. The bootstrap path now establishes PostgreSQL with pgvector, Redis, local storage, migrations, and worker startup expectations, but a PM or support person still has to infer appliance readiness from terminal output and scattered setup knowledge. Missing model/provider keys and failed background infrastructure can surface later as feature errors rather than as clear setup issues.

**Desired behavior:**
The appliance should expose a structured health/readiness contract that reports component-level status for the app, PostgreSQL, pgvector/RAG, Redis, document worker, drawing worker, storage, migrations, and model/provider configuration. Each component should report `healthy`, `degraded`, or `unhealthy`, with plain-language messages that help a PM or support person understand whether the appliance is ready and what needs attention. A simple setup/status UI should render the same information without requiring developer tools. Missing model keys, disabled provider configuration, worker/queue failures, unavailable pgvector/RAG, and migration/schema gaps should be presented as actionable setup/readiness issues instead of generic server failures.

**Key interfaces:**
- Health response contract - should include an overall status plus component statuses, messages, and enough diagnostic detail for support without leaking secrets.
- Component readiness checks - should cover database connectivity, pgvector/RAG schema availability, Redis connectivity, worker/queue readiness, local storage read/write capability, migration/schema readiness, and model/provider key configuration.
- App health/readiness endpoint - should keep machine-readable responses suitable for smoke automation and support tooling.
- Setup/status UI - should present the health response in plain language for PMs and support users.
- Model/provider configuration surface - should distinguish missing optional keys, selected provider keys that are required for configured features, and disabled/unconfigured providers.
- Worker/queue readiness surface - should make document-processing and drawing-extraction failures visible and actionable.

**Acceptance criteria:**
- [x] A health API returns structured component status for database, pgvector/RAG, Redis, workers, storage, migrations, and model/provider configuration.
- [x] The health response distinguishes healthy, degraded, and unhealthy states with plain-language messages.
- [x] A simple setup/status UI renders the health response without requiring developer tools.
- [x] Missing model keys or disabled provider configuration are shown as setup issues, not generic server errors.
- [x] Worker or queue failures are visible and actionable.
- [x] Tests cover at least healthy, missing Redis, missing model key, and unavailable RAG/pgvector states.

**Out of scope:**
- Backup and restore implementation.
- Starting, stopping, or repairing local services automatically from the UI.
- Replacing the existing bootstrap and smoke scripts, except where they should consume or align with the new health contract.
- Broad deployment redesign, Electron/Tauri packaging, or public SaaS launch work.
- Implementing typed RFI records or changing agent/workflow write behaviour.
