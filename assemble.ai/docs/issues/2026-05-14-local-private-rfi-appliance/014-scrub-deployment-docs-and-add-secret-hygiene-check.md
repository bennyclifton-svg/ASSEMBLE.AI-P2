## Agent Brief

**Category:** enhancement
**Summary:** Scrub deployment/setup documentation of live operational secrets and add a lightweight committed-secret hygiene check.

**Current behavior:**
The repo now has a truthful local/private bootstrap path and a strategy note that positions Sitewise as a local/private single-user project appliance. However, deployment and operational documentation still carries older SaaS/public-hosting assumptions and sensitive operational detail. The archived deployment guide includes a live-looking Supabase database connection string with an embedded password, server IP/root-login style instructions, domain/provider account details, and deployment commands that model passing `DATABASE_URL` as a Docker build argument. The Dockerfile also documents build-time `DATABASE_URL` usage, which risks baking secrets into image build history or generated artefacts.

Committed environment examples are mostly placeholder-based and useful, but the repo does not yet have a lightweight check that future contributors can run to catch obvious credential leaks in documentation and committed config. Local development defaults such as localhost database URLs should remain allowed when clearly scoped to local-only use.

**Desired behavior:**
Deployment and setup documentation should be safe to keep in the repo and aligned with the local/private appliance direction. Any live-looking credentials, passwords, private service endpoints, root-login details, account-specific dashboard details, and sensitive database URLs should be removed or replaced with placeholders and operator guidance. Docs should explain where real secrets belong at runtime, such as local `.env.local`, private service environment variables, or the operator's deployment secret store, without printing those values.

The Docker/deployment guidance should avoid recommending secret-bearing build arguments for server-only runtime values. If a build-time public value is genuinely required, it should be clearly distinguished from runtime secrets. The local/private setup docs and example env files should remain useful, but should only contain blank values, placeholders, or explicitly local-only credentials.

A lightweight hygiene check should scan committed documentation and config for high-confidence secret patterns and fail loudly when it finds live-looking values. The check should be documented so future contributors can run it before release or before handing the appliance to a customer. It should avoid noisy false positives for documented placeholders, intentionally blank values, localhost URLs, and clearly fake examples.

**Key interfaces:**
- Deployment and setup documentation - should use placeholders and current local/private appliance terminology rather than old public-SaaS-only or server-specific instructions.
- Docker build/runtime configuration guidance - should distinguish public build-time values from runtime secrets and should not model passing sensitive database URLs as build args.
- Environment example files - should remain a useful operator template while containing only blank, placeholder, or local-only values.
- Secret hygiene command or test - should scan committed docs/config for obvious database URLs with non-placeholder credentials, API key/token prefixes with live-looking values, private keys, webhook secrets, and account/server operational details.
- Contributor documentation - should state how to run the hygiene check and where real secrets should be configured instead.

**Acceptance criteria:**
- [x] Deployment and setup docs no longer contain live credentials, production passwords, sensitive database URLs, or account-specific operational access details.
- [x] Docs use placeholders and explain that real secrets belong in local `.env.local`, deployed service environment variables, or the operator's secret store.
- [x] Docker/deployment guidance no longer recommends passing sensitive server-only values such as `DATABASE_URL` as Docker build arguments.
- [x] The documentation aligns with the local/private appliance direction rather than old public-SaaS-only assumptions.
- [x] A lightweight script, test, or npm command scans committed documentation and config for obvious secret patterns and fails on high-confidence leaks.
- [x] The hygiene check allows intentional placeholders, blank env values, and explicitly local-only defaults such as localhost database URLs.
- [x] The check is documented so future contributors can run it before release or customer handoff.
- [x] Existing example env files remain useful without containing live secrets.

**Out of scope:**
- Rotating any real credentials that may have appeared historically.
- Designing a full production deployment platform, secret manager, or managed private-cloud release process.
- Rewriting the local bootstrap, health check, backup/restore, or RFI feature work beyond references needed for safe setup/deployment docs.
- Removing all historical archive material wholesale; only scrub or clearly neutralize sensitive and misleading operational detail.
- Adding runtime encryption, backup encryption, scheduled secret scanning services, or CI integration unless that is the smallest way to provide the lightweight check.
