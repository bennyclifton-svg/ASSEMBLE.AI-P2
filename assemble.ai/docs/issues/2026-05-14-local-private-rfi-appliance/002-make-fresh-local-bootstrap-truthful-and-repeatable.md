## Agent Brief

**Category:** enhancement
**Summary:** Make the local/private Sitewise bootstrap truthful and repeatable from a fresh checkout.

**Current behavior:**
Local setup is fragile and spread across several partial paths. The local service orchestration starts PostgreSQL with pgvector and Redis, but the database container still runs stale initialization SQL that can create tables with foreign keys before the referenced base tables exist. Main schema setup and RAG schema setup are separate commands, and the RAG setup path is not obviously migration-backed from a clean checkout. Local environment defaults do not clearly describe the complete appliance path, so the Next app, document worker, and drawing worker can rely on slightly different effective configuration. Local file storage fallback exists, but the setup docs and smoke path do not prove that storage, Redis, workers, the app, and RAG are all available after bootstrap.

**Desired behavior:**
A developer or private-appliance operator should be able to start from a clean checkout and follow one documented local/private setup path. That path should bring up PostgreSQL with pgvector, Redis, the Next app, the document processor worker, the drawing extractor worker, main schema setup, RAG schema setup, and local file storage without relying on stale SQL or undocumented manual ordering. The app and workers should load the same effective environment values. Required model/provider keys should be represented as clear placeholders so setup can proceed honestly even when optional AI features are not yet configured. The path should end with a smoke command or checklist that verifies the app, database, Redis, workers, storage, and RAG are reachable.

**Key interfaces:**
- Local service orchestration - should start PostgreSQL with pgvector and Redis without depending on stale table-creation SQL that conflicts with the application schema.
- Database setup flow - should initialize the main application schema and any auth-support schema needed by the app from a clean database.
- RAG setup flow - should initialize the pgvector-backed RAG schema through committed migrations or an explicit automated setup command.
- Environment loading contract - should define one effective local/private environment order shared by the Next app and both worker processes.
- Storage abstraction - should default to local filesystem storage when Supabase/cloud storage is absent or explicitly disabled.
- Smoke verification surface - should provide a command or checklist that checks app, database, Redis, workers, storage, and RAG readiness.
- Setup documentation - should describe the exact local/private startup sequence and required placeholder values for model/provider keys.

**Acceptance criteria:**
- [x] A clean local database can be initialized without stale foreign-key ordering failures from initialization SQL.
- [x] Main application schema setup is documented and either migration-backed or automated in the local/private bootstrap path.
- [x] RAG schema setup is documented and either migration-backed or automated in the local/private bootstrap path.
- [x] Local file storage is selected and usable by default when cloud storage is not configured.
- [x] Redis and both worker processes load the same effective environment as the Next app.
- [x] Setup docs describe the exact local/private startup sequence and required model-key placeholders.
- [x] A smoke command or checklist verifies that the app, database, Redis, workers, local storage, and RAG are all available after setup.

**Out of scope:**
- Backup and restore implementation.
- A user-facing appliance health dashboard beyond any smoke/readiness endpoint or command needed for this bootstrap.
- Secret-hygiene cleanup in deployment documentation, except where local setup docs would otherwise repeat unsafe examples.
- Production deployment redesign, Electron/Tauri packaging, or public SaaS launch work.
- Implementing typed RFI records or other contract-administration features.
