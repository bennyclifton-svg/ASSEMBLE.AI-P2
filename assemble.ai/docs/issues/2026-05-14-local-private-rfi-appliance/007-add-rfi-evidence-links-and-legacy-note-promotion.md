## Agent Brief

**Category:** enhancement
**Summary:** Let typed RFIs link to evidence and let legacy RFI notes be promoted without losing history.

**Current behavior:**
The product now has a typed, project-scoped RFI register with schema-backed records, numbering, status, priority, responsible party, due dates, list/detail/create/edit UI, API routes, and project refresh events. Legacy project notes still support `type='rfi'`, and notes can already have attached/transmittal documents. The wider project record also has documents, correspondence, and other artefact concepts, but typed RFIs do not yet expose a first-class evidence relationship. A PM cannot gather the documents, notes, or supported communication artefacts that justify an RFI onto the typed RFI itself, and existing legacy RFI notes remain stranded in the notes surface unless a user manually re-enters them as new typed RFIs. There is also no source-note link or idempotency guard that proves a promoted RFI came from a specific legacy note.

**Desired behavior:**
Typed RFIs should have first-class evidence links. A PM should be able to attach project documents to an RFI as evidence, view those links from the RFI register/detail experience, and remove or replace links without affecting the underlying documents. Where existing project record models support it, the same evidence relationship should also be able to point at an existing note or communication artefact. Evidence links should be scoped to the same project and organization as the RFI, validate that the target artefact exists, avoid duplicate links, and publish project refresh events after mutation.

Legacy RFI notes should be promotable into typed RFIs from a clear UI affordance or API action. Promotion should create or return a typed RFI whose core fields are derived from the note where possible, preserve a durable link back to the source note, and leave the original note intact. Running promotion more than once for the same source note should be idempotent or rejected with a clear existing-RFI response; it must not create duplicate typed RFIs. Promoted RFIs should appear in the typed RFI register and be managed like manually created RFIs, while still showing their source-note relationship for audit/history.

**Key interfaces:**
- `RfiRecord` / typed RFI persistence model - should gain evidence and source-note metadata without weakening existing numbering, status, priority, responsible-party, due-date, row-version, and soft-delete semantics.
- RFI evidence link model - should represent the RFI, project, organization, evidence target type, target id, optional label/context, created metadata, and uniqueness constraints that prevent duplicate links for the same target.
- RFI domain service - should own evidence-link validation, duplicate handling, project/organization scoping, source-note preservation, promotion idempotency, and project refresh emission through the existing event surface.
- Legacy note model - `type='rfi'` notes should be eligible for promotion; non-RFI notes should either be rejected or handled only when the product deliberately allows attaching them as evidence rather than promoting them.
- Document and communication artefact references - evidence linking should reuse existing project document and supported artefact records rather than creating a parallel file store.
- RFI API/UI contracts - should expose list/add/remove evidence behavior and a promote-from-note operation with clear validation errors and enough response data for the register/detail UI to refresh.
- Application Action Registry boundary - if evidence linking or promotion is exposed to agents/workflows, those writes must be registered actions with validation, preview/proposal behavior, audit, apply behavior, and project-event emission; no new mutating agent tool should bypass the registry.

**Acceptance criteria:**
- [x] A PM can link one or more project documents to a typed RFI as evidence, and the RFI detail/register experience shows those linked documents.
- [x] Evidence links validate project/organization scope and reject links to missing or out-of-project targets with clear errors.
- [x] Duplicate evidence links for the same RFI and target are prevented or handled idempotently.
- [x] A PM can link an existing note or supported communication artefact to an RFI where the existing data model can validate and display that target.
- [x] A legacy `type='rfi'` note can be promoted into a typed RFI from a clear UI affordance or API action.
- [x] Promotion preserves a durable source-note link and does not delete, overwrite, or hide the original note.
- [x] Duplicate promotion of the same source note is prevented or idempotently returns the existing typed RFI.
- [x] Promoted RFIs appear in the typed RFI register, can be edited/managed like manually created RFIs, and show enough source-note context for audit/history.
- [x] Evidence-link and promotion mutations publish project refresh events so open RFI views update without manual reload.
- [x] Tests cover document evidence linking, invalid/out-of-scope evidence targets, duplicate evidence handling, source-note preservation, duplicate promotion handling, and promoted-RFI register visibility.

**Out of scope:**
- Recording formal RFI responses, close/reopen lifecycle rules, or transition audit history.
- Exporting/issuing RFI artefacts, versioning issued outputs, or sending email.
- AI-drafted RFI workflows, approval cards for generated RFIs, and chat-driven promotion flows beyond respecting the action-only write boundary if any agent/workflow exposure is added.
- Full communication-artifact unification beyond the artefact types already safe to validate and display for this RFI slice.
- Bulk migration of all legacy RFI notes, deletion of legacy notes, or removal of the existing notes-based RFI capture path.
- Building typed registers for EOTs, defects, notices, site instructions, progress claims, or determinations.

## Implementation Notes

- Added `rfi_evidence_links` and `rfi_records.source_note_id` so typed RFIs can carry document, note, and correspondence evidence without creating a separate file model.
- Extended the RFI service with project/organization-scoped evidence validation, duplicate-link idempotency, evidence removal, and lightweight `type='rfi'` note promotion.
- Promotion is intentionally not a migration programme: it creates or returns a typed RFI for a single note, preserves the source-note pointer, links the note as evidence, and leaves the original note untouched.
- Added API routes for listing/adding/removing RFI evidence and promoting a single note into a typed RFI.
- Added register detail UI controls for adding/removing document, note, and correspondence evidence from newly created RFIs.
- Verified with focused service, API, and register UI tests.
