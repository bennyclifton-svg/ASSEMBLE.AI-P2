## Agent Brief

**Category:** enhancement
**Summary:** Add an approval-gated AI workflow that drafts typed RFI records from project context, asks only essential clarifying questions, and never writes before explicit approval.

**Current behavior after implementation:**
The product has a typed, project-scoped RFI register with schema-backed records, deterministic numbering, status, priority, responsible party, due dates, evidence links, legacy note promotion, response recording, close/reopen lifecycle rules, audit events, issued artefact versioning, project refresh events, API routes, UI surfaces, and focused service/API/UI tests.

The AI layer now raises typed RFI proposals through the registered action `correspondence.rfi.create` / tool `create_rfi`. The action validates typed fields, resolves responsible stakeholders and supported evidence targets in the current project, builds a user-readable approval preview, and creates the RFI plus evidence links only after approval. It does not send email, mark an RFI as issued, or fall back to a legacy `notes.type='rfi'` write.

**Desired behavior:**
When a PM asks the AI to draft or raise an RFI, the agent should read relevant project records, stakeholders, selected or retrieved documents, notes/correspondence where available, and RAG/knowledge context as needed. It should produce a proposed typed RFI containing user-readable fields such as title, question/request text, priority, responsible stakeholder or responsible-party label, due date, status, and evidence references. The proposal should cite the project documents or records that materially shaped the draft so the PM can check the basis before approval.

If the available context is enough to draft a useful RFI, the agent should create an approval-gated proposal without asking extra questions. If a required branch-setting fact is genuinely missing or ambiguous, the agent should ask one concise clarifying question and then continue from the answer. The workflow should not invent a recipient, due date, contractual fact, drawing reference, or evidence citation just to avoid asking.

Approving the proposal should create the typed RFI record and any supported evidence links through a registered action or registered workflow action steps. Rejecting the proposal should leave no typed RFI record or evidence links, while preserving the proposal/rejection in the approval/action invocation audit trail. The workflow must not send email, generate or mark an issued artefact, or bypass the Action Registry with a direct mutating agent tool.

**Key interfaces:**
- `RfiRecord`, `CreateRfiRequest`, `AddRfiEvidenceRequest`, and related status/priority/evidence types - the workflow should create schema-backed RFIs and link supported evidence targets rather than falling back to legacy note records.
- RFI domain service - should remain the boundary that owns numbering, project/organization scoping, responsible-party validation, due-date validation, evidence-link idempotency, row-version behavior, and project refresh assumptions.
- RFI create/propose action definition - add a registered action for creating a typed RFI proposal, with validation, preview/proposed diff, approval policy, apply behavior, action invocation audit, project event emission, and agent/workflow access. If evidence is created in the same approved operation, that must also be represented in the preview and action behavior.
- RFI drafting workflow - add a workflow/planner entry that gathers context, calls the model through an injectable/fakeable client, asks one branch-setting question only when needed, and materializes registered action proposal steps.
- Context assembly and retrieval - include current project facts, stakeholders, selected/relevant project documents, RAG excerpts, notes/correspondence evidence, and knowledge references where useful; keep project document/record citations in a structured form the preview can show.
- Approval and audit surfaces - approved proposals should resolve through the existing approval apply path; rejected proposals should resolve through the existing rejection path and be visible through the action invocation/approval audit trail.
- Agent tool catalog and action-only policy checks - expose any new agent capability as action-backed or workflow-backed; do not add a legacy mutating RFI tool or broaden the exemption inventory.

**Acceptance criteria:**
- [x] A PM can ask the AI to draft/raise an RFI from project context, selected/retrieved documents, or a concrete user instruction, and receives an approval-gated typed RFI proposal.
- [x] The proposal preview shows user-readable RFI fields, including title, request/question text, priority, responsible party where known, due date where known, status, and evidence references/citations.
- [x] Project documents or records that materially shape the recommended RFI content are cited in the proposal, and supported evidence targets are linked to the created RFI after approval.
- [x] If required proposal fields cannot be inferred safely, the agent asks one concise clarifying question instead of creating a weak proposal.
- [x] Approving the proposal creates the typed RFI only through registered action/workflow approval handling and emits the normal RFI project refresh event.
- [x] Rejecting the proposal creates no typed RFI record and no RFI evidence links, but leaves the approval/action invocation audit trail with the rejection.
- [x] The workflow never sends email, marks an RFI as externally issued, generates an issued artefact, or writes through a legacy mutating agent tool.
- [x] The action-only policy tests cover the new RFI action/workflow path and fail if it bypasses the Action Registry.
- [x] Tests use a fake model client or equivalent deterministic seam to cover read/context assembly, clarify-if-needed, proposal creation, approval/apply, rejection/no-write, evidence citation, and invalid/out-of-project evidence handling.

**Out of scope:**
- External email sending, mailbox integration, or provider-backed RFI issuance.
- Exporting/generated issued artefacts beyond reusing existing versioned export records if the UI already has them.
- Natural-language approval in chat; explicit approval controls remain required.
- Broad Correspondence specialist build-out, speculative contract-administration agents, or typed EOT/defect/site-instruction registers.
- Bulk legacy RFI note migration beyond using already-promoted/source-note context as evidence.

## Implementation Notes

- Added registered action `correspondence.rfi.create` with tool name `create_rfi`, exposed to the Design specialist as the action-backed path for AI-drafted RFIs.
- The action validates typed RFI fields, resolves project-scoped responsible stakeholders and evidence targets, deduplicates evidence references, and builds a user-readable approval preview before any record is created.
- Approval apply creates the typed RFI through the RFI domain service and links supported document/note/correspondence evidence only after approval.
- The action never sends email, marks the RFI as externally issued, or generates an issued artefact. Issued PDF/DOCX output remains a separate manual RFI export path.
- Agent prompt/routing guidance directs weekly/RFI drafting requests to action-backed tools and requires a concise clarifying question when required fields cannot be inferred safely.
- The action-only policy tests cover the new mutating action-backed tool path; focused action tests cover validation, preview, approved apply, evidence citation display, and out-of-project/missing evidence rejection.
