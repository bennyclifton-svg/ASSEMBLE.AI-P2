## Agent Brief

**Category:** enhancement
**Summary:** Add a verified backup/restore smoke path for local/private Sitewise project data and files.

**Current behavior:**
The local/private bootstrap path now brings up PostgreSQL with pgvector, Redis, workers, schemas, and local file storage, and the smoke path verifies those components are reachable. The app also has a storage abstraction with a local filesystem implementation and project records in PostgreSQL. What is still missing is a committed backup/restore contract that proves a local/private project can be backed up and restored with both database records and stored file payloads intact. At the moment, backup and restore are strategic requirements rather than a working, testable recovery path.

**Desired behavior:**
A developer or private-appliance operator should be able to create a backup for a Sitewise project from local/private database state and stored files, then restore it into a clean environment or clean project namespace without hand-editing SQL. The backup should be self-describing enough to identify the source project, application/schema version, creation time, included tables or record groups, and included file payloads. Restore should recreate the expected core project records and place file payloads where the storage abstraction can read them. The first format can be intentionally narrow, but its coverage and gaps must be documented so it is clear what is protected and what still needs later hardening.

**Key interfaces:**
- Backup command or service - should create a project-scoped backup artifact from PostgreSQL project records and local/private stored file payloads.
- Restore command or service - should load a backup into a clean environment or clean project namespace without manual SQL editing.
- Backup manifest format - should record project identity, backup creation time, app/schema version or migration marker, included record groups, included file payloads, and format version.
- Storage abstraction - restore should write files so existing document/download code can read restored file assets through the normal storage interface.
- Local/private smoke verification - should prove that a sample project can be backed up, restored, and inspected for expected records and files.
- Backup documentation - should state what this first backup format covers and what remains out of scope.

**Acceptance criteria:**
- [x] A project backup can be created from local/private storage and PostgreSQL state.
- [x] The backup artifact includes metadata for project identity, backup format version, app/schema version or migration marker, creation time, and included file payloads.
- [x] Restore can load the backup into a clean environment or clean project namespace without manual SQL editing.
- [x] A restored project contains its expected core project records and local files, and restored files are readable through the existing storage abstraction.
- [x] A scripted smoke check proves that a sample project can be backed up and restored successfully.
- [x] Documentation explains the first backup format's coverage, limitations, and any non-goals.

**Out of scope:**
- Cloud/Supabase storage backup and restore beyond keeping the interface compatible with future support.
- Encrypted, incremental, scheduled, or remote backups.
- Cross-version migration guarantees beyond recording the source app/schema version and failing clearly when unsupported.
- Full project export/import handoff UX or legal-grade archive packaging.
- Backing up unrelated infrastructure state such as Redis queues, model-provider secrets, analytics, or external service configuration.
