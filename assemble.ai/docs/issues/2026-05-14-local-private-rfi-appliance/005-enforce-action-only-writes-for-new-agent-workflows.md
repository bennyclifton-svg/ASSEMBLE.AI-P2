## Agent Brief

**Category:** enhancement
**Summary:** Add enforceable guardrails so new agent and workflow writes go through registered application actions.

**Current behavior after implementation:**
The codebase has an enforceable Action Registry policy for new agent and workflow mutations. Action definitions register stable action IDs, friendly tool names, actor policies, validation schemas, preview/proposal behavior, apply behavior, audit invocation rows, and project-event emission. Action-backed agent tools are exposed through the tool adapter, workflows materialize plans whose write steps point at registered action IDs, and tests fail when a new mutating tool or workflow write path bypasses the registry without an explicit legacy exemption.

**Desired behavior:**
From this point forward, new agent and workflow mutations should be action-only by default. A developer adding a mutating agent tool or workflow write step should either register it as an application action or add an explicit legacy exemption with a reason and migration target. The project should include a clear policy document and a CI-friendly test or static check that fails when a new mutating agent/workflow write path bypasses the Action Registry without that exemption. Existing legacy mutating tools may remain during migration, but they should be inventoried so the team can see which surfaces are intentionally grandfathered and where they should converge. Existing action-backed tools and workflows must keep their current approval-card behavior: validation, preview/diff, proposal/audit record, explicit approval, apply conflict handling, project-event emission, and workflow dependency behavior.

**Key interfaces:**
- `ActionDefinition` - the durable contract for registered actions, including input schema, actor policies, preview/proposal, apply/run behavior, emitted project events, and agent access.
- `defineAction()` and the action registry lookup APIs - should remain the source of truth for action IDs and exposed tool names.
- `AgentToolDefinition` - mutating tools should be detectable through their `mutating` flag and either resolve to a registered action or appear in the legacy exemption inventory.
- The action-to-agent adapter - should continue exposing friendly action-backed tool names without requiring agents to call generated action aliases directly.
- Workflow plan steps - mutating workflow steps should reference registered action IDs and fail clearly if an action is missing.
- Runner write-recovery and guard surfaces - the current regex-based guard list should be treated as frozen policy, not a place to add new write authority.
- Legacy mutating tool inventory - should capture tool name, domain, why it is exempt for now, and the intended migration target or retirement path.

**Acceptance criteria:**
- [x] A repo policy states that new agent and workflow mutating tools must be registered application actions unless explicitly listed as legacy exemptions.
- [x] A test or static check fails when a new mutating agent tool is registered without a matching action-registry entry or explicit legacy exemption.
- [x] A test or static check fails when a workflow write step references an unregistered action ID or uses a non-action write surface for new workflow behavior.
- [x] Existing legacy mutating tools are inventoried with exemption reasons and migration targets or retirement notes.
- [x] Action-backed friendly tools still resolve through the Action Registry and still produce approval cards with validation, preview, proposal/audit, apply, conflict handling, and project-event behavior.
- [x] Runner guard/recovery policy is documented as frozen: no new regex guard surfaces without a typed-schema/action-policy replacement plan.
- [x] Existing action registry, action dispatch, workflow runner, issue-variation workflow, and runner recovery tests still pass.

**Out of scope:**
- Migrating every existing legacy mutating tool onto registered actions in this issue.
- Migrating direct UI routes onto the Action Registry.
- Implementing typed RFI records or RFI workflows.
- Rewriting the runner or removing existing recovery guards.
- Adding new specialists, natural-language approval, or proactive background workflow behavior.
