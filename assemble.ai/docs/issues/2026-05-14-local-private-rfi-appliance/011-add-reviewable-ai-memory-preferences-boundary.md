## Agent Brief

**Category:** enhancement
**Summary:** Add a reviewable AI memory surface for project preferences and recurring context, with explicit precedence below schema records and documents.

**Current behavior:**
The current strategy now states the intended boundary: structured PostgreSQL records and stored files are canonical project truth, chat is interaction history, and AI memory is limited to editable preferences and recurring context. Agent prompts already receive assembled project context through the shared context assembly path, and workflow runs can snapshot operating preferences for auditability, but there is no PM-facing memory store for project preferences or recurring context.

There is an existing report/RAG memory concept for learned report table-of-contents or content patterns. That is not the same as PM-reviewable project memory, and recent report generation code already fences it away from overriding fixed report structure. Project UI preferences and organization defaults also exist, but those are configuration surfaces, not AI memory. As a result, useful recurring PM guidance either stays in chat history, lives in implementation settings, or has to be repeated manually.

**Desired behavior:**
A PM should have a small, reviewable memory area for the current project where they can add, edit, and remove entries such as drafting style, recurring project context, preferred assumptions, reporting tone, or PM-specific conventions. Each memory entry should be visibly categorized so it cannot be mistaken for a typed project record, document, issued artefact, RFI, risk, cost, programme, or stakeholder fact.

Only active or approved memory should be eligible for AI context. When included, memory should appear as a separate advisory section labelled as preferences/context, not as project facts. Context assembly and prompts should preserve a clear hierarchy: schema-backed records and stored document evidence take precedence over memory; memory takes precedence only over generic model assumptions or phrasing defaults. If memory conflicts with a project record or document-backed fact, the assembled context should either exclude the conflicting memory from the factual section or flag it as advisory/conflicting while still directing the model to rely on the schema/document source.

Manual UI edits should feel lightweight, but any AI-proposed or workflow-proposed memory changes must follow the action-only policy: registered action definitions, validation, preview, approval/audit where required, apply behavior, project event emission, and policy tests. The feature should leave a clean boundary for the later weekly briefing workflow to save accepted PM judgement as approved memory or as typed records, without implementing that full reporting workflow here.

**Key interfaces:**
- `AiMemoryEntry` or equivalent memory record - should be project-scoped, organization-scoped, distinguishable from project records, and carry type/category, content, active/approved state, source/actor metadata, and timestamps.
- AI memory repository/service - should own project and organization scoping, validation, listing, creation, editing, deletion/deactivation, and active-only retrieval for AI context.
- `ActionDefinition` and action registry APIs - should expose AI-proposed create/update/delete memory operations as registered actions if agents or workflows can mutate memory.
- `assembleContext()` and `AssembledContext` - should include approved memory through a separate prompt-ready field or clearly separated module section, with metadata sufficient to test inclusion and conflicts.
- Agent prompt builders - should present memory under an explicit advisory heading and state the precedence rule: records and documents override memory.
- Project event/audit surfaces - should refresh memory UI after changes and leave a trace for AI-proposed memory mutations.
- Existing report/RAG memory - should remain bounded to report pattern learning and must not become the hidden project-truth store for this feature.

**Acceptance criteria:**
- [x] A PM can open a project memory surface and view current entries, including clear empty, loading, and error states.
- [x] A PM can add, edit, and delete or deactivate memory entries for project preferences or recurring context.
- [x] Memory entries persist with project/organization scoping and visible type/category metadata.
- [x] The UI labels memory as preferences/context and clearly says it is not canonical project truth.
- [x] Approved or active memory can be included in AI context assembly as a distinct advisory section.
- [x] Deleted or inactive memory is not included in subsequent AI context assembly.
- [x] Schema facts and document evidence override memory when they conflict, with a focused test proving the model-facing context follows the canonical source.
- [x] Any agent or workflow path that creates, edits, or deletes memory is action-backed and covered by the action-only policy tests.
- [x] Tests cover memory add/edit/delete behavior, active-only context inclusion, and at least one conflict case where schema records or document evidence beat memory.
- [x] Existing report/RAG memory behavior remains scoped to report pattern learning and does not become a hidden project fact source.

**Out of scope:**
- Cross-project memory or precedent retrieval.
- Automatically mining chat history into memory without explicit review.
- The full weekly briefing/reporting workflow that decides whether accepted answers become records or memory.
- Replacing typed project records, RFI records, documents, issued artefacts, or audit logs with memory entries.
- Natural-language approval for memory writes.
- Broad prompt rewrite beyond the context and precedence changes needed for this boundary.
