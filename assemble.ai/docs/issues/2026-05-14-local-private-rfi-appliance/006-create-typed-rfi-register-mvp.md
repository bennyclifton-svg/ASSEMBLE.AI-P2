## Agent Brief

**Category:** enhancement
**Summary:** Add the first typed RFI register MVP so RFIs are schema-backed project records instead of only note-type prose.

**Current behavior:**
RFIs currently exist only as communication records with a generic record type. They can be titled, given freeform content, labelled as RFI, marked open or closed, dated, attached to documents through the note/transmittal path, and shown inside the notes/meetings/reports surface. That is useful as capture, but it is not a contract-administration register: there is no typed RFI schema, project-scoped RFI number sequence, responsible-party field, priority, due-date semantics, overdue display state, register filters, or reusable domain service that later RFI evidence, response, export, and AI workflow slices can build on. The current strategy identifies RFI as the first typed-record pilot, and the action-only write policy has landed, clearing this issue's blocker.

**Desired behavior:**
The product should have a first-class, project-scoped RFI register MVP. A PM should be able to manually create, list, view, and edit typed RFI records through normal UI and API flows. Each RFI should have a deterministic project-local number, title, request/question text, status, responsible party, due date, priority, timestamps, and enough versioning/update metadata to support later action/audit work. The register should show scan-friendly rows with number, title, status, responsible party, due date, priority where useful, and a derived overdue display state. The service should own the invariants: project/organization scoping, numbering, allowed statuses/priorities, due-date parsing, responsible-party validation, soft-delete or lifecycle assumptions, and update behavior. Create and update should publish project refresh events so open register views refetch without a manual reload.

This slice should make the manual register real without trying to complete the whole RFI lifecycle. It is acceptable for status to include values needed by the register filters, but response capture, close/reopen rules, audit history, evidence links, legacy-note promotion, exports, and AI-drafted RFIs belong to later issues.

**Key interfaces:**
- `RfiRecord` / typed RFI persistence model - should represent project, organization, RFI number or sequence, title, request text, status, priority, responsible party, due date, timestamps, and update metadata as structured fields rather than prose.
- RFI status and priority enums - should be explicit, validated, and shared between service, API, and UI. Overdue should be derived from due date plus non-terminal status, not manually typed as a stored status.
- RFI domain service - should provide create, update, get, and list/filter operations and should be the reusable boundary for later manual UI, registered actions, agent workflows, export, and legacy promotion.
- Responsible-party reference - should use existing project stakeholder concepts where available and return a human-readable display value for the register.
- RFI API contracts - should expose project-scoped list/create/update/get behavior with validation errors that are clear enough for the UI to surface.
- Project event channel - should include RFI create/update events or an equivalent typed-record event so register hooks can refetch after mutations.
- Register UI state and hooks - should support list, detail/create/edit flow, loading/error states, and filters for draft/open, overdue, responded, and closed views.

**Acceptance criteria:**
- [x] RFIs are stored in typed schema-backed records, not only as `notes` rows with an RFI type.
- [x] A PM can manually create an RFI with title, request/question text, status, responsible party, due date, and priority.
- [x] A PM can view and edit the core RFI fields through the UI, with validation feedback for invalid dates, statuses, priorities, or responsible-party references.
- [x] The register lists RFI number, title, status, responsible party, due date, priority where useful, and a derived overdue display state.
- [x] Filters support at least draft/open, overdue, responded, and closed views.
- [x] RFI numbering is deterministic and unique within a project, including after existing RFIs are already present.
- [x] Create and update operations publish project refresh events, and an open register view refetches after those events.
- [x] The domain service is covered for numbering, status/priority validation, due-date handling, responsible-party scoping, filtering, and update behavior.
- [x] API tests cover project scoping, create/list/get/update behavior, validation failures, and not-found/unauthorized cases.
- [x] UI tests cover at least one create/list/edit path and one filter/overdue display path.

**Out of scope:**
- AI-drafted RFI proposals, agent tools, approval cards, and RFI workflows.
- Recording formal RFI responses, close/reopen lifecycle rules, and transition audit history.
- Linking evidence documents or communication artefacts to typed RFIs.
- Promoting legacy RFI notes into typed RFI records.
- RFI export, issued artefact versioning, and email sending.
- Migrating every legacy RFI note or deleting the existing notes-based record surface.
- Building typed registers for EOTs, defects, notices, site instructions, progress claims, or determinations.
