## Agent Brief

**Category:** enhancement
**Summary:** Add reviewable/exportable RFI outputs and store each generated issued artefact as immutable version history.

**Current behavior after implementation:**
The product has a typed, project-scoped RFI register with numbering, status, priority, responsible party, due date, source-note promotion, evidence links, response recording, close/reopen lifecycle methods, lifecycle audit events, API routes, register UI, hooks, and focused service/API/UI tests. RFIs can now become controlled external-facing outputs through the RFI export routes and issued artefact service. Generated PDF/DOCX outputs are stored as immutable `rfi_issued_artefacts` versions backed by `file_assets` and storage, and the RFI detail UI shows the latest export plus prior versions.

The codebase has adjacent export patterns for notes, meetings, project reports, RFTs, addenda, TRRs, transmittals, and program PDFs. Those routes mostly generate a downloadable file on demand. That is not enough for this slice: generated RFI outputs need to be stored as project record history so regeneration creates a new version rather than replacing or silently re-rendering the previous issued artefact. There is also an `ai_artefacts`/`file_assets` store used for evaluation artefacts, but it is evaluation-oriented and does not currently model RFI issued-output versions.

**Desired behavior:**
A PM should be able to generate an export from a typed RFI and retrieve it later as part of the RFI's issued-output history. The exported artefact should include the project details, RFI reference/number, title, request text, responsible party, due date, linked evidence references, current status, and response details where useful for the current lifecycle. Generating the artefact should store a new immutable version with metadata such as version number, format, filename, file asset/storage reference, generated/issued timestamp, actor, source RFI row version, and content hash. Regenerating should append a new version and leave earlier versions available.

The RFI detail UI should show the latest generated artefact and a concise previous-versions list with download/retrieve controls. The implementation should not send email, mark correspondence as delivered through an external provider, or collapse this into generic note export. It should produce a stable, auditable project record output that later RFI action workflows can reuse.

**Key interfaces:**
- RFI issued artefact persistence - add a project/org-scoped model for generated RFI outputs, likely tied to `rfi_records`, `file_assets`, and storage; enforce monotonically increasing version numbers per RFI and preserve prior versions.
- RFI export data builder - collect project details, typed RFI fields, responsible party display, due date, evidence labels, status, response/date, and generated metadata into one stable export DTO.
- RFI export renderer - reuse existing PDF/DOCX export utilities where practical, but keep RFI-specific layout and content assertions in a dedicated module.
- RFI domain service - own generate/list/get latest/get version behavior, project/org validation, version-number allocation, source row-version capture, content hashing, storage write, and project refresh emission.
- RFI API routes - expose project-scoped generate and list/retrieve endpoints, with clear validation errors for missing RFIs, out-of-project access, unsupported formats, and storage failures.
- RFI register/detail UI and hook state - add controls for generating the export, show latest version metadata, and list/download previous versions without crowding the existing response/evidence/audit panels.
- Action Registry boundary - if this issue exposes "issue/export RFI" to agents or workflows, register it as an application action with preview/proposal behavior, audit invocation, apply logic, and event emission; do not add a direct mutating agent tool.

**Acceptance criteria:**
- [x] A PM can generate an RFI export from a typed RFI through the UI and API.
- [x] The exported artefact includes project details, RFI number/reference, title, request text, responsible party, due date, linked evidence references, current status, and response details where relevant.
- [x] Each generated artefact is stored with immutable version metadata including version number, generated timestamp, actor, format, source RFI row version, file/storage reference, and content hash.
- [x] Regenerating an artefact creates a new version and does not overwrite or delete prior issued history.
- [x] The RFI detail UI shows the latest generated artefact and previous versions with retrieve/download affordances.
- [x] Project/org scoping prevents reading or generating artefacts for RFIs outside the current project and organization.
- [x] Export/version mutations emit project refresh events so open RFI views update after generation.
- [x] Tests cover export content, version creation, regeneration without overwrite, prior-version retrieval, invalid/out-of-scope access, and UI display of latest plus previous versions.

**Out of scope:**
- Sending email or integrating with an external mail provider.
- AI-drafted RFI workflows, approval cards for generated RFI drafts, or chat-driven issuing beyond respecting the action-only boundary if exposed.
- Full issued-output unification across reports, transmittals, addenda, and TRRs.
- Bulk migration of legacy note exports into RFI issued-output history.
- Replacing the existing document `versions` table or general document repository versioning model.
