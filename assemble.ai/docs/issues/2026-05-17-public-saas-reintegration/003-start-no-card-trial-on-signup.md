## Agent Brief

**Category:** enhancement
**Summary:** Start a no-card public SaaS trial during signup using validated plan intent.

**Current behavior:**
Public pricing now links guests to registration with a Starter or Professional plan identifier in the URL. The registration page does not read that plan identifier, and the registration form submits only email, password, and display name to Better Auth. Better Auth user records currently include identity, display name, organization id, admin flags, and suspension state, but no trial fields. The signup hook already creates one organization/workspace and knowledge libraries for a new Better Auth user.

There is also legacy subscription/access code elsewhere that can still treat Free as a fallback state. For public SaaS signup, new trial state should be owned by the Better Auth user record and the shared public plan catalog, not by the legacy `users` table.

**Desired behavior:**
Registration should accept the intended public plan from the pricing URL, validate it against the shared public plan catalog, and start a 14-day no-card trial when signup succeeds. Only Starter and Professional are valid public trial plans. Missing plan intent should choose a conservative default from the public catalog, preferably Starter. Invalid plan intent should not silently create a trial against an unknown plan; the registration UI should surface a clear error or normalize to a documented default before signup.

The Better Auth user schema should store trial metadata directly on the auth-owned user record. Use explicit fields equivalent to `trialStartedAt`, `trialEndsAt`, `trialPlanId`, and `trialStatus`. The first status should represent an active trial. Trial dates should be calculated from the signup time using the trial duration in the shared plan catalog, not a duplicate hard-coded duration. The post-signup destination should be able to read the user's active trial state and intended plan through the existing authenticated user/session path or a small user/account API.

The existing one-user workspace creation behavior must remain intact: a successful signup creates exactly one organization/workspace for the user and associates that organization with the Better Auth user. This issue should not start paid checkout and should not enforce entitlements yet; it only records the trial state so the later entitlement issue can consume it.

**Key interfaces:**
- Public plan catalog - validates that requested trial plan is Starter or Professional and supplies the no-card 14-day trial duration.
- Registration page/form - reads plan intent from the URL, validates it, shows the selected plan or fallback, and submits it through the signup flow.
- Better Auth user schema - stores trial start, trial end, intended trial plan, and trial status on the auth-owned user record.
- Better Auth signup configuration and hooks - persists trial metadata and continues creating exactly one workspace/organization for the new user.
- Authenticated user/session read path - exposes enough trial state after signup for dashboard or later entitlement work to read active trial state and intended plan.

**Acceptance criteria:**
- [ ] Registration accepts Starter and Professional plan intent from pricing.
- [ ] Registration rejects or clearly normalizes invalid plan intent using the shared public plan catalog.
- [ ] Signup without plan intent uses a documented default public trial plan.
- [ ] Signup stores trial start, trial end, intended plan, and trial status on the Better Auth user record or an auth-owned schema.
- [ ] Trial end is calculated from the shared catalog trial duration.
- [ ] A new user still receives exactly one initial workspace/organization owned by that user.
- [ ] The post-signup authenticated user/session path can read active trial state and intended plan.
- [ ] New SaaS signup does not write or depend on legacy subscription fields as the trial source of truth.
- [ ] Tests cover signup with Starter, signup with Professional, signup without a plan, invalid plan intent, trial date calculation, and workspace creation.

**Out of scope:**
- Enforcing trial caps or expired-trial read-only behavior.
- Building the entitlement service or trial banner.
- Starting Polar checkout or creating a paid subscription.
- Wiring transactional email or email verification.
- Migrating existing users or local/private data into trial state.
- Removing legacy Free fallback behavior outside the signup path.
